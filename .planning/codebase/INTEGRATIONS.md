# External Integrations

**Analysis Date:** 2026-06-09

## APIs & External Services

**WordPress REST API:**
- Used for: registering and deleting webhook subscriptions via the Cinatra plugin's custom endpoint `cinatra/v1/webhooks`, listing existing subscriptions, reading instance metadata
- Client: host-internal lib `@/lib/wordpress-api` (resolved by monorepo); exposes `getWordPressAPISettings`, `readWordPressInstanceById`, `listWordPressWebhookSubscriptions`, `registerWordPressWebhookSubscription`, `deleteWordPressWebhookSubscription`
- Auth: WordPress application passwords (stored in monorepo host configuration, accessed via `getWordPressAPISettings()`)
- Referenced in: `src/settings-page.tsx`

**WordPress MCP Adapter (WordPress/mcp-adapter plugin):**
- Used for: registering each configured WordPress site as a parallel MCP server for Cinatra agents
- Probe helper: `probeWordPressInstanceMcpAdapter` from host-internal `@/lib/wordpress-mcp-connection`
- Endpoint resolver: `resolveWordPressMcpEndpoint` from `@/lib/wordpress-mcp-connection`
- Status values: `registered`, `not_installed`, `auth_error`, or unreachable
- Referenced in: `src/settings-page.tsx` (`WordPressMcpAdapterSection` component)

## Data Storage

**Databases:**
- Not applicable — this connector holds no database of its own

**Widget Auth Config:**
- Stored and read via host-internal helpers `generateWidgetAuthConfig` / `readWidgetAuthConfig` from `@/lib/wordpress-widget-auth`
- Contains: `apiKey`, `webhookSecret`, `generatedAt` fields
- Storage mechanism determined by monorepo host (file-based or DB-backed; not visible in this repo)

**File Storage:**
- Not applicable

**Caching:**
- Next.js path revalidation via `revalidatePath` is called after every mutating server action to bust the Next.js cache for `/connectors/cinatra-ai/wordpress-assistant-connector/setup`

## Authentication & Identity

**Extension Authorization:**
- All server actions and the page render gate on `requireExtensionAction("@cinatra-ai/wordpress-assistant-connector", "manage")` from `@cinatra-ai/sdk-extensions`
- This enforces that only users with `manage` rights on this connector can view or mutate credentials and webhooks

**WordPress Plugin Auth (API key):**
- The connector generates an `apiKey` that the `cinatra.php` WordPress plugin sends as `Authorization: Bearer <key>` when calling back into Cinatra
- The `webhookSecret` is used to sign outbound webhooks with `X-Cinatra-Sig-256`
- Both are generated by `generateWidgetAuthConfig()` and displayed for copy-paste into the WordPress plugin settings

## Webhooks & Callbacks

**Incoming (Cinatra receives from WordPress):**
- Endpoint: `/api/webhooks/wordpress` on the Cinatra host application
- Trigger: WordPress publishes a post (`post_published` event type)
- Registered per WordPress instance via the connector's settings page (`registerWordPressWebhookSubscription`)
- Signature verification: `X-Cinatra-Sig-256` signed with `webhookSecret`

**Outgoing (Cinatra calls into WordPress):**
- WordPress REST API calls to `cinatra/v1/webhooks` endpoint for subscription management
- WordPress MCP adapter probed at the resolved MCP endpoint per instance

## Environment Configuration

**Required env vars:**
- `NEXT_PUBLIC_APP_URL` — public base URL of the Cinatra app; used to build the `/api/webhooks/wordpress` callback URL registered with WordPress
- `BETTER_AUTH_URL` — fallback when `NEXT_PUBLIC_APP_URL` is absent (defaults to `http://localhost:3000`)

**Secrets location:**
- No `.env` file in this repo
- WordPress credentials (site URL, application password) are stored and retrieved by the monorepo host through `@/lib/wordpress-api`
- Widget `apiKey` and `webhookSecret` are stored by `@/lib/wordpress-widget-auth` (monorepo-controlled)
